# ipTIME N702E Firmware Analysis

본 프로젝트는 ipTIME N702E 공유기 펌웨어에 포함된 웹 서버 바이너리(`httpd`)를 정적 분석하여, 공유기의 로그인 인증 방식과 인증 처리 흐름을 분석한 내용이다.

## 1. 프로젝트 개요

- **분석 대상**: ipTIME N702E 펌웨어 (버전: 12.102)
- **분석 목표**: 웹 서버의 인증 로직 역추적 및 보안 메커니즘 분석
- **사용 도구**: Ghidra

## 2. 분석 환경 및 구성

프로젝트 디렉터리는 다음과 같이 구성하였다.

- `firmware/`: 분석 대상 펌웨어 원본 바이너리
- `guide.md`: 펌웨어 획득 및 추출 과정 정리 문서
- `images/`: 분석 과정에서 사용한 이미지 자료

## 3. 핵심 분석 내용

### 3.1 인증 관련 핵심 함수 식별

펌웨어를 Ghidra로 분석한 결과, `Symbol Tree`의 `Exports` 항목에서 인증 로직과 관련된 핵심 함수들을 식별할 수 있었다.

문자열 기반 탐색만으로는 전체 인증 흐름을 정확히 파악하기 어려웠지만 Exported function 목록을 기준으로 접근함으로써 인증 처리에 직접 관여하는 함수들을 보다 효율적으로 찾을 수 있었다.

![추출함수](images/exported_functions.png)

식별한 주요 함수는 다음과 같다.

- `auth_authorize`
- `auth_check_userpass`
- `check_password_cache`

이 함수들을 중심으로 호출 관계와 내부 동작을 추적하여 인증 흐름을 분석하였다.

### 3.2 핵심 함수 분석

먼저 `auth_check_userpass` 함수와 내부에서 호출되는 `check_password_cache` 함수의 동작을 확인하였다.

아래는 Ghidra의 디컴파일 결과 일부이다.

~~~c
bool auth_check_userpass(void)
{
  int iVar1;
  undefined4 in_a3;
  
  // 패스워드 캐시 확인 및 자격 증명 검증
  iVar1 = check_password_cache(); 
  
  // 검증 실패 시 시스템 로그 기록
  if (iVar1 != 0) {
    syslog_msg(1, "{17}", in_a3);
  }
  
  // 인증 결과 반환
  return iVar1 != 0;
}
~~~

해당 함수는 단순히 사용자 입력 비밀번호를 비교하는 역할에 그치지 않고 인증 실패 시 `syslog_msg`를 호출하여 시스템 로그를 남기는 동작도 수행한다.

즉, `auth_check_userpass`는 다음 두 가지 역할을 동시에 수행한다.

1. 자격 증명 검증 결과 확인
2. 인증 실패 시 로그 기록

이를 통해 인증 로직이 단순 비교 함수가 아니라 실패 로그를 시스템 수준에서 추적할 수 있도록 설계되어 있음을 확인할 수 있었다.

### 3.3 인증 메커니즘 심층 분석

이후 `auth_authorize` 함수를 분석하여, 웹 서버가 클라이언트의 인증 요청을 처리하는 전체 흐름을 확인하였다.

~~~c
// HTTP 헤더의 'Authorization' 값이 'Basic '으로 시작하는지 확인
iVar2 = strncasecmp(pcVar4, "Basic ", 6);
if (iVar2 == 0) {
  // Base64로 인코딩된 인증 정보 디코딩
  base64decode2(&DAT_004296ec, pcVar4 + 6, 0x100);
  
  // 디코딩된 문자열에서 아이디와 비밀번호를 구분하는 ':' (0x3a) 위치 탐색
  pcVar4 = strchr(&DAT_004296ec, 0x3a);
  if (pcVar4 != (char *)0x0) {
    *pcVar4 = '\0'; // ':'를 널 문자로 치환하여 문자열 분리
    
    // 분리된 ID와 PW를 통해 최종 인증 수행
    iVar2 = auth_check_userpass(&DAT_004296ec, pcVar4 + 1, puVar3, param_1 + 0x464);
    
    if (iVar2 == 0) return 1; // 인증 성공 시 정상 처리
    goto LAB_00410454;        // 인증 실패 시 401 Unauthorized 호출
  }
}
~~~

분석 결과, `auth_authorize` 함수는 HTTP Basic Authentication 방식으로 전달된 인증 정보를 다음과 같은 순서로 처리한다.

#### 인증 처리 흐름

1. `Authorization` 헤더가 `Basic ` 스키마로 시작하는지 확인
2. 헤더의 Base64 인코딩 데이터를 디코딩
3. 디코딩된 문자열에서 `:` 문자를 기준으로 사용자 ID와 비밀번호를 분리
4. 분리된 값을 `auth_check_userpass`에 전달하여 최종 검증 수행
5. 검증 성공 시 요청을 승인하고, 실패 시 `401 Unauthorized` 응답을 반환

#### 분석 포인트

- **인증 파싱**  
  `strncasecmp`를 통해 `Basic ` 스키마 여부를 확인한 뒤, Base64 디코딩을 수행한다.

- **문자열 분리**  
  `strchr`를 사용하여 `:` 문자의 위치를 찾고, 해당 위치를 NULL 문자로 치환하여 ID와 비밀번호를 각각 독립된 문자열로 분리한다.

- **최종 인증 검증**  
  분리된 사용자 입력값은 `auth_check_userpass`로 전달되며, 여기서 실제 자격 증명 검증이 수행된다.

- **인증 실패 처리**  
  인증이 실패할 경우 인증 실패 로그가 기록되며, 이후 `401 Unauthorized` 응답이 반환된다.

위 분석을 통해 해당 펌웨어의 웹 인증이 **HTTP Basic Authentication**을 기반으로 구현되어 있으며

인증 정보 파싱 → 계정/비밀번호 분리 → 자격 증명 검증 → 실패 시 로깅 및 401 응답 반환이라는 흐름으로 구성되어 있음을 확인하였다.

## 4. 분석 결과 요약

이번 분석을 통해 ipTIME N702E 공유기 웹 서버의 인증 로직이 다음과 같은 구조로 동작함을 확인하였다.

- 클라이언트는 `Authorization: Basic ...` 헤더를 통해 인증 정보를 전달한다.
- 서버는 Base64 디코딩 후 `ID:PW` 형식 문자열을 분리한다.
- `auth_check_userpass` 및 `check_password_cache`를 통해 자격 증명을 검증한다.
- 인증 실패 시 시스템 로그를 남기고, `401 Unauthorized` 응답을 반환한다.

즉 해당 인증 로직은 비교적 전형적인 Basic Authentication 처리 구조를 따르며 인증 실패에 대한 로깅 기능까지 포함하고 있다.

## 5. 느낀점

이번 프로젝트를 통해 공유기 펌웨어 내 웹 서버 바이너리를 직접 추적하면서 단순한 로그인 기능도 내부적으로는 여러 함수 호출과 문자열 처리 과정을 통해 동작한다는 점을 확인할 수 있었다.

특히 기존에는 리버싱 문제 풀이 중심으로 학습해 왔지만, 실제 펌웨어를 대상으로 핵심 바이너리를 식별하고 인증 흐름을 분석해 보면서 정적 분석이 실환경 소프트웨어에 어떻게 적용되는지 체감할 수 있었다. 그리고 당연한거지만 내가 연습해왔던 문제파일들보다 훨 복잡하고 보기 힘들었다.

또한 인증 로직, 문자열 처리, 로그 기록 방식 등을 함께 확인하면서 펌웨어 분석이 단순한 함수 해석을 넘어 시스템 보안 구조를 이해하는 과정이라는 점을 배울 수 있었다.
